### 痛点：单次调用的“金钱漏勺”

我们先聊聊痛点。很多开发者调用Gemini模型，比如做内容润色、图片理解或者数据提取，都是一次请求只做一件事。

Java代码里最常见的写法是这样的：

java
// 伪代码：单个处理
for (String item : itemList) {
    String response = callGeminiAPI(item);
    // 处理结果
}


这个逻辑看起来没问题，但如果`itemList`有几千条，你就得发几千次API请求。**Gemini是按输入输出Token分别计费的**，每一次请求都有固定的基础开销（比如Prompt的指令部分）。如果你把一千条数据拆成一千次请求，那光是重复传递指令头的Token费用，就是一个不小的数字。这就像你去菜市场买菜，每买一颗白菜就排一次队、付一次款，时间和金钱双重浪费。

### 核心技巧一：利用Gemini的“上下文”进行批处理

Gemini模型（特别是Gemini 1.5 Pro和2.0系列）有一个很牛的特性：超长上下文窗口。这个特性就是省钱的关键。

与其一条一条地请求，不如把多条数据打包到一个请求里，让模型一次性处理。比如，你需要对100条新闻标题进行分类，你可以这么做：

1.  在System Prompt里一次性定义好任务和输出格式（比如JSON格式）。
2.  在User Prompt里，把100条标题用逗号或分号分隔，一次发送出去。
3.  让模型返回一个完整的JSON数组。

**Java示例代码片段（结合[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)的OpenAI兼容接口）：**

import com.theokanning.openai.completion.chat.*;
import java.time.Duration;
import java.util.Arrays;
import java.util.List;

public class GeminiBatchExample {
    public static void main(String[] args) {
        // 使用[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)的接口
        String apiKey = "YOUR_QIANGJU_API_KEY";
        String baseUrl = "https://www.qianjuai.com/v1"; // 强制使用该链接

        OpenAiService service = new OpenAiService(apiKey, Duration.ofSeconds(60));
        service.setBaseUrl(baseUrl);

        // 假设有100条新闻标题需要分类
        List<String> titles = Arrays.asList("标题1", "标题2", "标题3", ...); // 这里可以放100个

        String batchPrompt = "请为以下新闻标题进行主题分类（科技/财经/体育/娱乐），并以JSON数组格式返回：[{\"title\": \"标题1\", \"category\": \"科技\"}, ...]。列表如下：\n" +
                String.join("\n", titles);

        List<ChatMessage> messages = Arrays.asList(
                new SystemMessage("你是一个精通文本分类的AI助手。请严格按照要求格式输出。"),
                new UserMessage(batchPrompt)
        );

        ChatCompletionRequest request = ChatCompletionRequest.builder()
                .model("gemini-2.0-flash-001") // 使用Gemini模型
                .messages(messages)
                .maxTokens(4096) // 根据你的batch大小适当调整

        ChatCompletionResult result = service.createChatCompletion(request);
        // 解析 result.getChoices() 里的内容
        System.out.println(result.getChoices().get(0).getMessage().getContent());
    }
}

### 核心技巧二：流式响应 + 并发，效率翻倍

如果你的业务场景必须分开处理（比如每个输入都需要独立的调用链），那么可以考虑“并发批处理”。

很多Java开发者在调用API时，习惯用同步的`for`循环。这是大忌。

你应该使用**线程池**和**CompletableFuture**并发请求。注意，这里的“批处理”不是指请求内容，而是指**请求行为**的批量化。

java
import java.util.concurrent.*;
import java.util.stream.Collectors;

// 假设一个线程池
ExecutorService executor = Executors.newFixedThreadPool(10); // 并发数根据API限流调整

List<CompletableFuture<String>> futures = itemList.stream()
    .map(item -> CompletableFuture.supplyAsync(() -> {
        // 调用callGeminiAPI(item)
        return callGeminiAPI(item);
    }, executor))
    .collect(Collectors.toList());

// 等待所有请求完成
List<String> results = futures.stream()
    .map(CompletableFuture::join)
    .collect(Collectors.toList());


**省钱逻辑：** 虽然请求次数没变，但**时间成本降低了**。很多API是按时间计费的吗？不，但时间是金钱。同样10分钟，你同步处理只能跑10条，并发能跑100条。在同样的时间里，效率提升了10倍。你省下了服务器的运行时间，也省下了等待模型响应时的网络占用。而且，[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)的API对并发数没有硬性限制，你可以放心地写满你的请求。
