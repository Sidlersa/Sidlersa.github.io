## 首先，我们得搞清楚“不兼容”到底长什么样

在开始正式测试前，我先列出我遇到的几种典型的“伪兼容”表现，你可以在自己的项目里对比排查一下：

1. **模型ID被悄悄替换了**：你传了 `gpt-5.2-pro`，平台直接把它映射成了 `gpt-3.5-turbo`，然后给你生成一段质量极低的内容。
2. **响应格式错乱**：接口返回的JSON结构不是标准的 `choices[0].message.content`，而是自定义字段，导致解析直接抛 KeyError。
3. **流式输出（Stream）直接断流**：SSE流在前端卡死20秒后自动断开，没有任何错误消息。
4. **Token计数完全失真**：返回的 `usage.total_tokens` 与实际消耗对不上，导致开发者后台统计混乱。

以上这四类“伪兼容”问题，在我这次测试中，有5个库无一幸免。

## “伪兼容”陷阱的高峰期，如何自保？

结合我这次的测试经验，我总结了3条实操守则，建议你在接入任何“GPT-5.2pro兼容”服务时，逐条验证：

### 守则1：一定要测试流式输出

很多伪兼容服务在非Stream模式下能糊弄过去，但在Stream模式下一碰就碎。用下面这段代码验证一下，10秒内你就能看出端倪：

python
from openai import OpenAI
client = OpenAI(
    base_url="https://www.qianjuai.com/v1",
    api_key="你的千聚API密钥"
)
stream = client.chat.completions.create(
    model="gpt-5.2-pro",
    messages=[{"role": "user", "content": "说一句关于AI的名言"}],
    stream=True
)
for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="")


如果上面的代码在你选的库里，流在中间断开、或者只输出一个空字符串，那么恭喜你，你遇到了“伪兼容”。

### 守则2：检查Token计费是否与官网一致
