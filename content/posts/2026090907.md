## Java 开发的四大避坑指南

光有代码还不够，在实战中，你可能还会踩到这几个隐藏得很深的坑。我把它们总结出来，帮你一次性避开。

### 1. 别用错默认 API 版本
很多 Java 库默认请求的是 `gpt-3.5-turbo` 或更旧的模型。你明明想用 Groq 的 Llama 3，但因为模型名没写对，结果被路由到了一个更便宜但性能弱的模型上，或者干脆报错。
**解决方案**：在千聚的官方文档里找到模型的确切名称（例如 `groq-llama3-70b-8192`），在代码中明确指定 `model` 参数。

### 2. 忽略“提示词长度”的成本炸弹
很多低代码框架在后台会自动拼接大量系统提示词和上下文。你发给 API 的“有效请求”可能比你想象的大得多。
**解决方案**：在 Java 代码中，手动控制 `max_tokens` 和 `messages` 数组的总长度。如果对话很长，记得做截断或总结，只保留关键上下文。

### 3. 高并发下的“锁”陷阱
如果你用同步方式并发请求 API，并且没处理好连接池，很容易触发平台的限流（429错误）。
**解决方案**：使用 `OkHttp` 或 `Apache HttpClient` 做连接池管理，或者直接用异步客户端。在千聚的“纯 AZ”分组上，并发稳定性会更高，但成本略贵（官方×1.5），你需要平衡。

### 4. 忽视“模型分组”的隐性成本
千聚里有很多分组，比如“默认（混合）”和“限时特价”。如果你把高成本模型放在默认分组里，它的人工智能路由可能不会自动帮你走最便宜的路线。
**解决方案**：KISS原则。如果你明确要省钱，就在代码中锁死 `base_url` 指向的 `限时特价` 分组对应的 URL，而不是用默认混合分组。这能让你对自己的每一分钱都了如指掌。
