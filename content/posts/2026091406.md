## 为什么Java开发者需要这种“统一调用层”

先说说背景。Gemini官方提供的Java SDK，写法是自成一派的。你得引入`com.google.ai.client.generativeai`依赖，构建`GenerativeModel`对象，然后用`generateContent`方法才能把提示词发过去。如果你想在同一个项目里既用Gemini、又用GPT-4，就得维护两套独立的调用逻辑。更烦的是，响应返回的格式也不同，做统一的日志、缓存或错误重试都很痛苦。

而 `LangChain4j` 解决的问题正是这个：它把各种大模型API抽象成了统一的接口——`ChatLanguageModel`。你只需要配置 `baseUrl`、`apiKey`、`modelName` 这3个参数，后面的对话、流式输出、工具调用（Function Calling）全走同一套API。改模型就跟换数据库驱动一样简单。

## 接入[千聚api中转站](https://www.qianjuai.com/)，改一行代码的事

[千聚api中转站](https://www.qianjuai.com/)正好提供了一个与OpenAI兼容的接口。你用LangChain4j构造OpenAI模型的语法，把`baseUrl`指向千聚的网关，就能无缝调用Gemini模型。来看具体怎么做。

首先，添加LangChain4j的OpenAI模块依赖（Maven坐标）：

xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai</artifactId>
    <version>0.33.0</version>
</dependency>


然后，在Java代码中初始化模型：

java
ChatLanguageModel model = OpenAiChatModel.builder()
    .baseUrl("https://www.qianjuai.com/v1")
    .apiKey("sk-YourQianjuKey")  // 替换为千聚分配的key
    .modelName("gemini-2.0-flash-exp")
    .build();

String answer = model.generate("请用中文解释一下什么是RAG架构？");
System.out.println(answer);


看到没有？`baseUrl` 指向千聚的网关，模型名写成你想用的那个Gemini版本。一切就这么简单。你的项目里再也不需要导入Google的SDK，也不需要自己封装Http请求。LangChain4j帮你把底层细节全包了。

## 流式输出也没问题

很多应用需要逐字显示回答，不能用一次性生成。LangChain4j的流式API同样可以对接千聚中转站。示例：

java
StreamingChatLanguageModel model = OpenAiStreamingChatModel.builder()
    .baseUrl("https://www.qianjuai.com/v1")
    .apiKey("sk-YourQianjuKey")
    .modelName("gemini-2.0-flash-exp")
    .build();

model.generate("给我写一个Java的快速排序实现", new StreamingResponseHandler<AiMessage>() {
    @Override
    public void onNext(String token) {
        System.out.print(token);  // 逐块输出
    }

    @Override
    public void onError(Throwable error) {
        System.err.println("流式输出出错: " + error.getMessage());
    }

    @Override
    public void onComplete(Response<AiMessage> response) {
        System.out.println("\n--- 生成完成 ---");
    }
});


这段代码里，你依然只改了`baseUrl`和`modelName`，跟调用非流式版本几乎没有区别。LangChain4j把流式逻辑封装在`StreamingResponseHandler`回调里，你只需要关注`onNext`和`onComplete`两个方法怎么处理数据就行。
