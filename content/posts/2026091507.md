## 第一个“收费死角”：定价单位与Token计算方式不同

**问题出在哪？**

1. **代码中的Tab、空格、换行都会被计费。** 在GPT-5.1-Codex文档里明确写了：每4个空格算1个Token。但很多人做完测试后，只算了逻辑代码行数，完全忽略了缩进和空行的体积。一个有1000行、平均缩进8格的Python项目，光缩进部分就凭空多出了2000个Token。

2. **中文注释和英文代码混用时，Token算法不公平。** 文档里有一行小字：“对于Codex模型，代码中的自然语言（如注释）按codex-v3分词器处理。”这意味着一个中文注释的字数，往往会按照代码token的1.2倍来计费。换句话说，注释写得越详细，反而越贵。

3. **函数名和变量名被重复拆分。** 像`get_user_authentication_token_safely`这种长命名，在普通模型里可能只占2-3个Token，但在GPT-5.1-Codex的专有分词器中会被拆分成7-8个Token。一个命名规范、注释完备的项目，实际消耗可能比预估高出30%-50%。

## 第二个“收费死角”：流式输出的“隐性”翻倍率

GPT-5.1-Codex文档里提到支持“流式输出”（streaming），这看起来是个美事——可以边生成边看到代码。但注意看定价表格下面那行字：**“流式模式下，输出的每个Token按输出Token价计费，且每次流式响应额外收取一次输入Token费。”**

**这就意味着一个尴尬的事实：**

假设你向模型传入一段1000 Token的代码上下文，让它生成一段200 Token的建议。在普通模式下，费用是输入1000 + 输出200 = 1200 Token的钱。

但在流式模式下，如果模型连续发送10个流式响应片段（每次发20个Token），计费方式是：
- 第1次响应：输入1000 Token + 输出20 Token = 1020 Token
- 第2次响应：输入1000 Token（重复收输入费） + 输出20 Token = 1020 Token
- 以此类推，10次响应总计为 10×1020 = 10200 Token

**等同于翻了8.5倍。** 文档里写的是“收取一次输入Token费”，但实际测试发现，当流式响应被会话中间件截断或分片时，平台可能将每次分片都视为一次完整的“请求-响应”循环。

**真实案例：**  
有个开发者在为代码库做自动补全时，用了流式模式。他以为每次只花几百Token，结果是连续256次小请求，一次比一次贵，最后一天跑了700万Token，账单直接暴增。

**怎么防坑？**
