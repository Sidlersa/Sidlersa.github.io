## 第二个烧钱参数：`max_tokens` 设置过高，直接烧穿预算

这个参数是我踩过最大的坑。我当时图省事，直接照搬了 OpenAI 示例代码里的 `max_tokens=4096`。

问题来了：Claude 的上下文窗口非常大（比如 Sonnet 有200K），如果你把 `max_tokens` 设置得过高，而你的 Prompt 又很复杂，Claude 会拼命“思考”怎么填满这4096个Token的输出。很多情况下，它只需要输出几十个Token的简短答案，但因为设置了4096，它就会额外“添油加醋”，输出一堆无用的废话，而这些废话，全是按量计费的！

**正确的做法：**

1.  **对于简单问答**：设置 `max_tokens=500` 或 `1000` 就足够了。
2.  **对于代码生成**：根据预估代码量，设置 `max_tokens=2048` 左右。
3.  **对于长文本分析**：可以设置高一点，但也要有上限，比如 `max_tokens=4096`。

别让 API 替你浪费钱，精确控制输出长度，是省钱的第一步。

## 第三个烧钱参数：`temperature` 参数过小，导致“无效循环”

你可能觉得 `temperature` 参数只是控制创意程度，跟成本没关系。错了，关系大了去了。

在 Claude 中，如果 `temperature` 设得太低（比如0.1），模型会倾向于选择概率最高的“最安全”的回复。但有时候，这种“确定性”会导致模型陷入一种“钻牛角尖”的状态，反复尝试生成同一个答案，从而引发不必要的重复计算和Token消耗。

**我的经验值：**

- **分析、翻译等严谨任务**：`temperature` 设为 0.5 - 0.7。
- **创意写作、头脑风暴**：`temperature` 设为 0.8 - 1.0。
- **代码生成**：`temperature` 设为 0.2 - 0.5，避免产生随机变量名。

不要低于 0.1，那通常只会让你的钱包受伤，模型效率变低。

## 第四个烧钱参数：超时和重试机制设置不当，浪费大量请求

这又是一个“隐形杀手”。默认的超时时间通常很长（比如 120 秒），而且重试次数可能是无限次。

当你的 Prompt 有问题，或者网络波动导致请求失败时，你的客户端会傻傻地等上两分钟，然后重试。一次不行，重试两次、三次……这期间，每一次重试都会产生新的计费（取决于 API 的账单策略，有些按请求次数，有些按失败后的 token 消耗）。我上个月就因为这个，白白多了好几百块钱的请求费。

**我的血泪教训：**

- **设置合理超时**：对于 Claude 的对话模型，**建议设为 30-60 秒**。如果超过这个时间还没回应，大概率是出问题了，直接断开，重新构造请求。
- **限制重试次数**：设置 **最多重试 3 次**。如果连续3次失败，直接给用户报错，而不是盲目无限重试。
- **开启指数退避**：如果重试，每次等待的时间要逐次增加（比如 1秒 -> 2秒 -> 4秒），避免瞬间造成大量并发请求，导致更严重的阻塞。
