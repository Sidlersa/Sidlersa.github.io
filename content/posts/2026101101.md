## 野路子四：把重复的 RAG 链路做成“函数调用”，减少冗余 Token

很多 RAG 应用的 Token 消耗浪费在重复的上下文传递上。比如你的系统在回答前总要先生成 3 个不同的检索 query，再把文档内容和 query 一起发回模型——这是最烧钱的。

**怎么操作：**

利用千聚API支持的 `Function Calling` 机制，把检索、评分、选择这些重复步骤封装成函数输出，模型只返回函数名和参数，不回复自然语言，极大减少输出 Token。

举个例，你的 RAG 在生成最终答案前，需要先“总结文档”，你可以让模型只返回 `{"function": "summarize_docs", "params": {...}}`，而不是长篇大论一段话。

这样，模型输出 token 数直接从 500 降到 20，千聚API的计费是按 token 算的，你算算能省多少。
