### 问题根源：为什么你的账单会暴涨？

我们先简单拆解一下典型的 API 账单构成。

**1. 无用 Token 浪费**
用户输入的历史消息、系统提示词写得太啰嗦、或者每次请求都带着完整的上下文。GPT-4 的输入 Token 计费比输出贵，如果你每次发 4000 个 Token 的上下文，其中 3000 个是重复的历史对话——那就是在烧钱。

**2. 高费率模型被低价值任务占用**
一个简单的翻译请求，你调用了 `gpt-4-turbo`，而不是 `gpt-4o-mini`。前者价格是后者的几十倍。更常见的是，很多人默认代码里写死 `model=gpt-4`，从没想过给不同任务分配不同模型。

**3. 请求高峰期的无脑重试**
服务一挂，代码自动重试 10 次，每次还带相同请求，结果网关直接把你的并发额度打满，产生超额费用。

**4. 缺乏本地缓存**
同一个向量化请求、同一个 prompt 的补全，你每天跑了 1 万次。其实缓存一次就能解决。

### 省钱改造第二步：实现智能路由 + 提示词压缩

这是最核心的一步。我们不再让代码所有请求都发向同一个昂贵模型，而是写一个网关层函数，根据任务类型和 prompt 长度动态选择模型。

#### 代码示例：分层定价模型路由

python
def get_best_model(prompt: str, task_type: str = "chat") -> str:
    """
    根据任务重要性和prompt长度，自动选择性价比模型。
    task_type 可以是 'chat', 'translate', 'summarize', 'code_gen'
    """
    # 短 prompt、低价值任务 -> 用小模型
    if task_type in ["translate", "summarize"] and len(prompt) < 200:
        return "gpt-4o-mini"
    # 中等任务 -> gpt-4o-mini 足矣
    elif task_type == "code_gen" and len(prompt) < 1500:
        return "gpt-4o-mini"
    # 复杂推理、长上下文 -> 降级到 4o 而不是4
    elif task_type == "chat" and len(prompt) > 2000:
        return "gpt-4o"
    # 高精度任务才用4-turbo
    else:
        return "gpt-4-turbo"


**关键点**：`gpt-4o-mini` 在千聚上的价格是官方价格的 0.6 倍左右。你用 `4o-mini` 替代 `4-turbo`，成本直接降低 一个数量级。

#### 代码示例：Prompt 自动压缩

很多模型本身就支持 `max_tokens` 控制，但更实用的省钱方法是从源头减少输入。

python
def compress_prompt(messages: list, max_context_length: int = 2000) -> list:
    """
    移除无意义的历史消息，只保留系统提示和最近的用户回复。
    """
    system_msg = [msg for msg in messages if msg["role"] == "system"]
    recent_msgs = [msg for msg in messages if msg["role"] != "system"][-3:]  # 只留最新的3轮
    return system_msg + recent_msgs

### 省钱改造第三步：构建请求缓存层

对于大量重复的查询，比如“今天天气怎么样”或者“翻译一个固定短语”，没必要每次都请求大模型。我们在网关层加一个内存缓存。

#### 代码示例：基于 Prompt 哈希的缓存

python
import hashlib
import json
from functools import lru_cache

@lru_cache(maxsize=128)
def cached_chat_completion(prompt: str, model: str) -> str:
    """
    根据prompt和model的组合哈希，返回缓存结果。
    注意：此实现简化了参数，实际请考虑 temperature 等。
    """
    messages = [{"role": "user", "content": prompt}]
    response = client.chat.completions.create(model=model, messages=messages)
    return response.choices[0].message.content

def smart_chat(user_input: str):
    # 先尝试命中缓存
    model = get_best_model(user_input)
    try:
        result = cached_chat_completion(user_input, model)
        return result
    except Exception:
        # 缓存失败则走正常流程
        pass


**注意**：缓存适用于结果确定的场景（翻译、格式化、生成固定文案），不适合需要创造性的聊天。

### 省钱改造第四步：控制重试与异步批处理

高峰时段，避免“无限重试”。使用指数退避，且重试时自动降级到更便宜的模型。

#### 代码示例：退避重试 + 模型降级

python
import time

def request_with_retry_and_downgrade(messages, model, max_retries=2):
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                request_timeout=10
            )
            return response
        except Exception as e:
            print(f"请求失败: {e}, 第{attempt + 1}次重试")
            if attempt == max_retries - 1:
                # 最后重试降级到最便宜的模型
                print(f"降级到 gpt-4o-mini")
                return client.chat.completions.create(
                    model="gpt-4o-mini",
                    messages=messages,
                    request_timeout=15
                )
            time.sleep(2 ** attempt)  # 指数退避
    return None


**效果**：你不再为失败请求而全额支付昂贵的模型费用，避免了无谓损耗。
