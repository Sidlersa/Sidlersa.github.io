## 坑三：不控制上下文长度，导致Token爆炸

这是最隐蔽的坑。很多人跑R1的时候，习惯把对话历史越积越长，每次调用都把以前的所有对话内容带进去。R1会不断“反思”这些内容，导致一次性推理就能吃掉几千甚至上万Token。

**解决方案：** 在每次请求中，精简上下文，并且使用system prompt控制思考深度。

保持你的用户输入尽量短，不要把之前的全部对话都塞进去。或者在每次调用中只用当前的用户消息，以及一个简短的system prompt来限定模型的行为。

看这个优化后的示例：

python
import tiktoken  # 用来估算Token数
import openai

client = openai.OpenAI(
    api_key="sk-your-qianju-key",
    base_url="https://www.qianjuai.com/v1"
)

# 将用户输入精简一下
user_content = "帮我分析这个Python代码的时间复杂度：for i in range(n): for j in range(n): print(i+j)"

# 计算输入Token数，避免超长
enc = tiktoken.get_encoding("cl100k_base")
input_tokens = len(enc.encode(user_content))
print(f"输入Token估计：{input_tokens}")

response = client.chat.completions.create(
    model="deepseek-r1",
    messages=[
        {"role": "system", "content": "你是一个代码分析专家。对每个问题请只输出分析结果，不要进行多余的说明和反复修正。"},
        {"role": "user", "content": user_content}
    ],
    max_tokens=200,   # 限制输出长度，防止模型反复修正
    temperature=0.3,  # 降低随机性，减少废话
    top_p=0.9         # 控制采样范围
)

output_tokens = response.usage.completion_tokens
total_cost = (input_tokens + output_tokens) / 1_000_000 * 0.7  # 假设0.7元/百万Token
print(f"这次请求花费约：{total_cost:.4f}元")


通过tiktoken估算输入Token，再通过限制max_tokens和调整温度参数，你能把单次请求的成本控制在几分钱甚至更少。
