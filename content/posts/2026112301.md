## 为什么账单会失控？

先别急着骂中转站。DeepSeek R1本身的定价是透明的，但API调用过程中的各种“隐形消耗”才是真正的账单刺客。

- 请求失败后的无限重试
- 高重复率的无效token消耗
- 流式输出与非流式输出之间的巨大浪费
- 模型参数配置过于保守（比如温度设得太低导致重新生成）
- 缓存机制没开启，每次请求都重新算

这些才是天价账单的真正制造机。

## 3个隐藏设置，改了直接省一半

### 1. 关闭“无脑重试”，锁定 max_retries

很多中转站的 SDK 默认会设置较高的重试次数（如 3-5 次），一旦遇到网络抖动或模型限流，自动重试会重复消耗你的 Token。

**解决方案：**
在调用 `openai.Completion.create` 或 `client.chat.completions.create` 时，明确指定 `max_retries=0` 或 `max_retries=1`。

示例代码：
python
import openai

client = openai.OpenAI(
    base_url="https://www.qianjuai.com/v1",
    api_key="YOUR_API_KEY",
    max_retries=1
)

response = client.chat.completions.create(
    model="deepseek-r1",
    messages=[{"role": "user", "content": "你好"}]
)


这个改动最直接——减少无效请求，单次任务调用次数就降下来了，总Token消耗自然跟着降。

---

### 2. 开启“缓存命中”，省掉重复计算

DeepSeek R1 这种推理模型，响应时间比普通模型长，Token消耗也大。如果你在不同对话里问了高度相似的问题，每次模型都要从头算一遍，费用翻倍。

**解决方案：**
在请求 API 时，使用缓存参数（部分中转站或官方 SDK 支持），或者在配置层开启服务端缓存。[千聚ai大模型聚合站](https://www.qianjuai.com/)（www.qianjuai.com）的官方接口默认支持智能缓存，你只需要确保请求的 `user` 参数是唯一的，同时避免在每次请求中带上随机值破坏缓存命中。

如果中转站支持自定义缓存策略，优先开启。一个高重复率的任务（比如翻译、摘要、批量问答），缓存开启后能省 30%-50% 的 Token。

---

### 3. 使用“流式输出”（streaming），而不是一次性返回

DeepSeek R1 的完整输出长度可能很长，一次性返回（非流式）会占用更大的上下文窗口和 Token 计数。流式输出（streaming）不仅用户体验好，而且对账单更友好——Token 消耗会按实际生成量逐步释放，而不是一次性把所有输出算进去。

**解决方案：**
代码里加上 `stream=True`：

python
response = client.chat.completions.create(
    model="deepseek-r1",
    messages=[...],
    stream=True
)


这样不仅响应速度快了、延迟低了，而且账单更可控。你还能在流式输出的过程中，判断输出内容是否够用，提前中断（比如输出到一定字数就够了），避免模型“过度解释”多输出一堆无效Token。
