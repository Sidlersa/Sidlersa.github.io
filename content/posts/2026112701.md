## 示例一：裸用 RestTemplate 直连 OpenAI 端点

这是网上出现频率最高的一种写法：直接在 Spring Boot 项目里随便配个 RestTemplate，把请求发给 `https://api.openai.com/v1/chat/completions`。代码看上去很简洁：

java
RestTemplate restTemplate = new RestTemplate();
HttpHeaders headers = new HttpHeaders();
headers.setBearerAuth("sk-xxx");
HttpEntity<String> request = new HttpEntity<>("{\"model\":\"o3-mini\",\"messages\":[{\"role\":\"user\",\"content\":\"Hello\"}]}", headers);
ResponseEntity<String> response = restTemplate.postForEntity("https://api.openai.com/v1/chat/completions", request, String.class);


这段代码放在本地测一次没问题，但拿到生产环境里跑，问题就会马上暴露：RestTemplate 默认没有连接池，每次请求都要新建 TCP 连接，响应慢且连接不稳定；直接写死 `application/json` 请求体，很容易把大段 context 一起发出去，Token 数量控制不住；而且国内网络环境直连 OpenAI 端点延迟很高，重试策略也没有设置，失败率会高出很多。

**优化方案**

改用 `OkHttp` / `HttpClient5` 加连接池，同时把终点改为国内可直接直连的中转站 `https://www.qianjuai.com/v1`。

java
// 使用 OkHttpClient + 连接池
OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(new ConnectionPool(10, 5, TimeUnit.MINUTES))
    .connectTimeout(30, TimeUnit.SECONDS)
    .readTimeout(60, TimeUnit.SECONDS)
    .build();

// 构造请求体（限制上下文大小）
String body = "{\"model\":\"o3-mini\",\"messages\":[{\"role\":\"user\",\"content\":\"Hello\"}],\"max_tokens\":200}";

Request request = new Request.Builder()
    .url("https://www.qianjuai.com/v1/chat/completions")
    .header("Authorization", "Bearer your_qianju_key")
    .post(RequestBody.create(body, MediaType.get("application/json; charset=utf-8")))
    .build();


关键优化点：

- 连接池复用：减少握手开销，单次请求延迟下降 40% 以上
- 明确的超时时间：o3-mini 推理耗时本身不低，但流式场景下 `readTimeout` 不能太短，建议不低于 60 秒
- 限制 max_tokens：o3-mini 上下文最长能到 200k，但实际业务中大多数场景不需要这么多，限到 200 ～ 500 就能省下一大半 Token

## 示例二：手动拼接 Streaming 输出

另一个非常容易踩坑的地方是流式响应。很多人都知道 o3-mini 支持 SSE 流式输出，但写出来的代码往往是这样的：

java
InputStream inputStream = httpResponse.getEntity().getContent();
BufferedReader reader = new BufferedReader(new InputStreamReader(inputStream));
String line;
StringBuilder fullResponse = new StringBuilder();
while ((line = reader.readLine()) != null) {
    fullResponse.append(line);
}
System.out.println(fullResponse.toString());


这个代码有两个问题：把所有的 SS​E 事件混在一起当作普通文本去读取，不仅解析会丢数据，而且客户端要在流完全结束后才能拿到输出，失去了流式输出的核心优势——边生成边显示；另外每次 flush 到客户端的是一整段字符串，如果网络稍有抖动就会重读整个包，Token 实际消耗和展示的内容完全对不上。

**优化方案**

使用官方推荐的 `Jackson` 或 `Gson` 逐行解析 SSE 流，并实时推送。

java
// 使用 OkHttp EventSource 或自建 SSE 解析
client.newCall(request).enqueue(new Callback() {
    @Override
    public void onResponse(Call call, Response response) throws IOException {
        try (BufferedReader reader = new BufferedReader(response.body().charStream())) {
            String line;
            while ((line = reader.readLine()) != null) {
                if (line.startsWith("data: ") && !line.equals("data: [DONE]")) {
                    String json = line.substring(6);
                    ChatCompletionChunk chunk = new ObjectMapper().readValue(json, ChatCompletionChunk.class);
                    String delta = chunk.getChoices().get(0).getDelta().getContent();
                    if (delta != null) {
                        // 直接推送至客户端，如 WebSocket 或 SSE
                        webSocketSession.sendMessage(new TextMessage(delta));
                    }
                }
            }
        }
    }
});


关键优化点：

- 逐行解析 `data:` 前缀，并实时推送 delta 内容，客户端能在生成第一个 token 后立即看到
- 正确检测 `[DONE]` 标识来终止流
- 不需要一次性把整段 response 组装完再输出，token 消耗和网络带宽都会大幅节约
